제조 AI는 Prediction 문제가 아니다

2026.05.28 · Manufacturing AI · Cause-Effect Mapping · Data Validation

제조 AI에서 중요한 것은 단순히 높은 accuracy가 아니다.

현재 데이터가 실제 공정의 원인과 결과를 신뢰성 있게 설명하고 있는지를 먼저 검증해야 한다.

많은 AI 프로젝트는 입력(X)과 결과(Y)를 연결하는 문제로 설명된다.

좋은 feature를 만들고, 좋은 모델을 선택하면 더 높은 accuracy를 얻을 수 있다는 방식이다.

하지만 제조 AI에서는 데이터가 존재한다고 해서 원인과 결과가 자동으로 연결되지는 않는다.

실제 제조 환경에서는:

  • 어떤 공정 조건이
  • 어떤 샘플에 적용되었고
  • 어떤 품질 결과로 이어졌는가

자체가 불명확한 경우가 많다.

즉 제조 AI의 핵심은 단순 prediction이 아니라,

Cause → Effect Mapping

을 검증하는 문제에 가깝다.

이번 프로젝트에서도 모델 개발 이전에 가장 먼저 해야 했던 일은, 모델 선택이 아니라 데이터의 cause-effect mapping을 검증하는 과정이었다.

왜 제조 AI에서는 원인과 결과 연결이 어려운가

프로젝트의 목표는 coining 압력을 기반으로 bump 품질(FBH/FBD)을 예측하고, 목표 품질을 만족하는 압력을 추천하는 AI 모델을 만드는 것이었다.

핵심은 단순 prediction이 아니었다.

모델은:

  • 이전 상태(RBH)
  • 공정 조건(Coining Pressure)
  • 최종 결과(FBH/FBD)

사이의 관계를 학습해야 했다.

즉:

"어떤 상태에서 어떤 압력을 가했더니 어떤 결과가 나왔는가"

를 설명할 수 있어야 했다.

문제는 제조 데이터가 생각보다 단순하지 않다는 점이었다.

같은 제품 모델이라도:

  • LOT마다 초기 상태가 다르고
  • 이전 공정 특성이 다르며
  • 공정 편차와 변동성이 존재한다.

즉 같은 압력을 적용하더라도, LOT에 따라 결과가 달라질 수 있다.

따라서 제조 AI에서는 단순히 데이터를 많이 모으는 것이 아니라,

현재 데이터가 실제 동일 샘플의 공정 흐름을 설명하는가

를 먼저 검증해야 했다.

실제 프로젝트에서 마주친 문제

1. 검사 통과 = 모든 bump가 정상은 아니었다

프로젝트 초기에는 다음과 같은 가정을 하고 있었다.

"검사를 통과한 bump는 모두 spec-in이다"

하지만 실제 데이터를 확인해보니 그렇지 않았다.

SPC fail이 존재하더라도, coining 압력 문제가 아니라고 판단되면 pass로 전환되는 경우가 있었다.

즉 검사 통과 LOT이라도, 수십만 개의 bump 중 일부는 다른 원인으로 비정상일 수 있었다.

이 문제는 단순 데이터 정제 문제가 아니었다.

평가 기준 자체가 실제 현장의 품질 판정 방식과 일치하지 않는다는 의미에 가까웠다.

결국:

제조 AI에서는 label 자체가 현실 공정을 올바르게 설명하는지부터 검증해야 했다.

2. 동일 LOT이라고 같은 샘플은 아니었다

다음 문제는 공정 조건과 검사 결과를 실제로 연결할 수 있는가였다.

프로젝트에서는:

  • 이전 공정(MBM/SPP)
  • Coining 공정
  • 검사 결과(FBH/FBD)

를 연결해서 분석해야 했다.

특히 목표는:

RBH → Coining Pressure → FBH/FBD

관계를 학습하는 것이었다.

하지만 제조 환경에서는 단순히 LOT ID가 같다고 해서, 반드시 동일한 공정 흐름이라고 볼 수는 없었다.

실제 생산에서는 하나의 LOT 내부에도 여러 strip/quad가 존재했고, 각 strip은 서로 다른 상태와 공정 이력을 가질 수 있었다.

따라서 단순 LOT 단위 연결만으로는:

  • 이전 상태
  • 적용 압력
  • 최종 품질 결과

가 실제 동일 샘플을 의미한다고 보장할 수 없었다.

이 상태에서 데이터를 그대로 사용하면, 모델은 실제 공정의 인과관계가 아니라 서로 다른 샘플이 섞인 상관관계를 학습하게 된다.

3. 압력 데이터가 무엇을 의미하는지도 먼저 정의해야 했다

문제를 더 복잡하게 만든 것은, 현재 확보 가능한 압력 데이터의 의미였다.

검사 공정은 한 번만 수행되지 않는다.

실제 생산에서는:

  • 초도 검사
  • 재초도 검사
  • 추가 재검사

등이 반복적으로 진행될 수 있다.

하지만 MES에 저장된 압력값은, 모든 재검사 과정의 전체 압력 이력이 아니라

"최종적으로 생산에 사용된 압력"

에 가까웠다.

또한 strip 공정에서는, MBM/SPP 또는 Coining 단계에서 검사를 통과하지 못한 strip은 폐기되었다.

즉 fail된 strip은 이후 공정 흐름으로 이어지지 않는다.

결국 현재 데이터 구조에서:

"어떤 압력을 가했더니 어떤 결과가 나왔는가"

를 분석하려면,

  • 실제 생산까지 이어졌고
  • 동일 strip/quad 기준으로 연결 가능하며
  • 검사 통과 이후의 품질 결과로 해석 가능한 데이터

를 중심으로 분석해야 했다.

즉 단순히 데이터가 존재하는지가 아니라,

현재 데이터가 실제로 무엇을 의미하는가

를 먼저 정의하는 과정이 필요했다.

그래서 모델보다 먼저 검증해야 했던 것들

결국 모델 학습 이전에 먼저 해야 했던 것은, “쓸 수 있는 데이터”의 기준을 정의하는 일이었다.

우리는 다음 조건들을 먼저 검증했다.

  • 동일 strip/quad 기준으로 공정 흐름이 연결되는가
  • 이전 상태(RBH)와 최종 결과(FBH/FBD)가 동일 샘플 기준으로 연결되는가
  • 해당 압력이 실제 생산 압력으로 해석 가능한가
  • 검사 결과가 실제 생산 승인 데이터를 의미하는가
  • 현재 label이 실제 현장의 품질 판정 기준을 반영하는가

이 조건을 만족한 데이터만 이후 분석과 모델 학습에 사용했다.

제조 AI는 모델링 이전에 공정을 설명할 수 있어야 한다

많은 제조 AI 프로젝트들은 model accuracy에 집중한다.

하지만 실제 제조 환경에서 중요한 것은:

  • 현재 데이터가 실제 공정을 설명하는가
  • 모델이 실제 인과관계를 학습하고 있는가
  • 결과를 공정 관점에서 해석할 수 있는가

이다.

특히 제조 데이터는 여러 시스템에 분산되어 있고, 공정 흐름과 운영 기준이 현실 환경에 따라 계속 변한다.

따라서 제조 AI에서는 모델링 이전에, 현재 데이터가 실제 공정의 cause-effect relationship을 설명할 수 있는지 먼저 검증해야 한다.

어쩌면 제조 AI에서 가장 어려운 것은, 좋은 모델을 만드는 일이 아니라

"현재 데이터가 실제 공정을 설명하는 데이터인지 판단하는 일" 일 수도 있다.
